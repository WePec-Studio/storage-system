# Go 命名规范

## 避免包名重复 (stuttering)

当类型名或函数名包含包名时，必须去掉重复前缀，遵循 Go 惯用风格：

```go
// ✓ 好的示例
storage.Object
storage.OpWrite
storage.InitCollection()

// ✗ 不好的示例
storage.StorageObject
storage.StorageOpWrite
storage.StorageInitCollection()
```

从 Nakama 借鉴类型时，若原名带有 `Storage` 前缀（如 `StorageObject`、`StorageOpWrite`），转换到本项目时去掉该前缀。

# Go 代码注释规范

编写任何 Go 代码注释时，必须遵循以下规范

## 基本原则

- 注释使用中文
- 只用半角标点符号（`,` `.` `:` `()` 等），不使用全角标点
- 句末不加句号
- 英文单词、数字、代码片段与中文字符之间前后各留一个空格
- 注释只携带代码逻辑本身的描述，不携带任何任务/计划编号（如 `#2`、`Step 2.1`）

## 标点与排版示例

```go
// Redis 缓存数据过期时间，单位小时         ✓
// Redis缓存数据过期时间，单位小时           ✗ 英文前后缺少空格

// 若参数为空则返回 nil 并打印错误日志       ✓
// 若参数为空则返回nil并打印错误日志。        ✗ 缺少空格，且末尾有句号

// 默认缓存时间为 24 小时                   ✓
// 默认缓存时间为24小时。                   ✗ 缺少空格，且末尾有句号
```

---

## 函数 / 方法注释

### 核心原则

函数注释关注**行为和返回值**，不描述实现细节：

```go
// ✓ 好的示例：关注行为
// Sort 将数据按 Less 方法定义的升序排列，对 Less 和 Swap 的调用次数为 O(n*log(n))

// ✗ 不好的示例：暴露实现细节
// Sort 使用快速排序算法对数据进行排序，首先选取基准元素...
```

例外：性能关键场景可以注明算法复杂度。

### 模板

```go
// FuncName 一句话概括函数的核心作用（动宾结构，简明扼要）
//
// 可选补充说明（架构设计、依赖组件、特殊机制等，1~3 句）
//
// 参数：
//   - paramName: 参数含义及约束
//   - paramName2: 参数含义及约束
//
// 返回值：
//   - ReturnType: 返回内容说明；异常情况下的返回值
//
// 注意：（可选，仅在有重要副作用或使用限制时填写）
//   补充使用注意事项
func FuncName(param Type) ReturnType {
```

### 各部分说明

**摘要行（必填）**

- 格式：`// FuncName + 动宾短语`
- 简洁，不超过 20 字
- 示例：`// NewAutoCacheManager 创建一个自动缓存管理器`

**补充说明（可选）**

- 与摘要行之间留一个空行
- 适合描述架构层级、依赖组件、特殊行为

**参数（有参数时必填）**

- 每个参数占一行，格式：`//   - paramName: 说明`
- 说明合法性约束、默认行为

**返回值（有返回值时必填）**

- 每个返回值占一行，格式：`//   - Type: 说明`
- 明确边界/异常情况下的返回值

**功能说明（仅复杂函数使用）**

- 只在函数行为较复杂、仅靠摘要行无法说清时添加
- 描述行为和结果，不描述实现步骤
- 示例：`// 若 key 不存在则自动从数据源加载并写入二级缓存`

### 完整示例

```go
// NewAutoCacheManager 创建一个自动缓存管理器
//
// 包含二级缓存 (Redis) 和本地内存缓存
//
// 参数：
//   - keyPrefix: 缓存 key 的前缀，不能为空
//
// 返回值：
//   - IAutoCache[Key, Val]: 自动缓存接口实例，若参数非法则返回 nil
func NewAutoCacheManager[Key cacheKey, Val any](keyPrefix string, opts ...Option) IAutoCache[Key, Val] {
```

## 变量 / 字段注释

单行注释，简洁说明含义及单位/约束：

```go
// Redis 缓存数据过期时间，单位小时
cacheTimeOutHour time.Duration

// 本地内存缓存过期时间，单位秒
memTimeOutSecond time.Duration

// 是否启用自动清理
autoClear bool
```

---

## 不需要注释的情况

- 命名已足够自解释的简单变量和函数
- 显而易见的逻辑，如 `// i 是循环变量`
- 实现细节（用代码本身表达，而非注释）

# Git Commit 规范

基于 [约定式提交 v1.0.0](https://www.conventionalcommits.org/zh-hans/v1.0.0/)

---

## 提交信息结构

```
<类型>[可选范围]: <描述>

[可选正文]

[可选脚注]
```

---

## 类型

| 类型 | 说明 |
|------|------|
| `feat` | 新增功能 |
| `fix` | 修复 bug |
| `docs` | 文档变更 |
| `style` | 代码格式调整（不影响逻辑，如缩进、空格） |
| `refactor` | 代码重构（不新增功能，不修复 bug） |
| `perf` | 性能优化 |
| `test` | 新增或修改测试用例 |
| `build` | 构建系统或依赖变更 |
| `ci` | CI/CD 配置变更 |
| `chore` | 其他非业务性改动 |
| `revert` | 还原之前的提交 |

---

## 规则

**描述**

- 紧跟在类型/范围之后，冒号和空格后直接书写
- 简短概括本次变更，不超过 72 个字符
- 不以句号结尾

**范围（可选）**

- 用圆括号包裹，描述影响的模块或范围
- 示例：`feat(auth):`, `fix(cache):`

**正文（可选）**

- 与描述之间空一行
- 说明变更的原因、背景或对比，不描述实现细节
- 每行不超过 72 个字符

**脚注（可选）**

- 与正文之间空一行
- 格式：`令牌: 值` 或 `令牌 #值`
- 令牌使用 `-` 连字符，如 `Reviewed-by`、`Refs`

---

## 破坏性变更

有两种标记方式，选其一即可：

**方式一：在类型后加 `!`**

```
feat(api)!: 移除用户登录接口的 password 字段
```

**方式二：在脚注中声明**

```
feat(api): 调整用户登录接口结构

BREAKING CHANGE: password 字段已移除，请改用 token 认证
```

---

## 示例

**仅有描述**

```
docs: 修正 README 中的拼写错误
```

**包含范围**

```
feat(cache): 新增二级缓存自动刷新支持
```

**包含正文**

```
fix(cache): 修复并发请求下缓存击穿问题

引入请求 ID 机制，忽略非最新请求的响应结果，
同时移除已过时的超时兜底逻辑
```

**破坏性变更**

```
feat(auth)!: 移除 BasicAuth 支持

BREAKING CHANGE: BasicAuth 已废弃，请迁移至 JWT 认证方式
```

**还原提交**

```
revert: 还原用户登录接口的结构调整

Refs: 676104e, a215868
```

---

## 与语义化版本的对应关系

| 提交类型 | 版本影响 |
|----------|----------|
| `fix` | PATCH 版本升级 |
| `feat` | MINOR 版本升级 |
| `BREAKING CHANGE` | MAJOR 版本升级 |

---

## 注意事项

- 一次提交只做一件事，如果变更涉及多种类型，拆分为多次提交
- 类型统一使用小写
- `BREAKING CHANGE` 必须大写
- 提交信息中不得出现任何 Claude 相关信息，包括但不限于 `Generated by Claude`、`Co-authored-by: Claude`、`Claude` 等
